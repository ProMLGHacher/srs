# Спецификация production-ready решения: WebRTC-комнаты на SRS

Документ описывает целевое **production-ready** решение на основе текущего прототипа: React-клиент, Node.js (Express) как шлюз сигналинга и медиа-прокси к **SRS 5** (WHIP/WHEP, SFU), WebSocket для координации пиров в «комнатах».

**Версия документа:** 1.0  
**Базовый прототип:** репозиторий `srs` (ветка/состояние на момент составления).

---

## 1. Назначение и границы системы

### 1.1. Продукт

Сервис **групповых аудио-видео комнат** в браузере: участники заходят в комнату по идентификатору, публикуют локальный медиапоток (WHIP → SRS) и подписываются на потоки других участников (WHEP). Сигнальный сервер сообщает, кто сейчас в эфире, чтобы клиенты открывали нужные WHEP-сессии.

### 1.2. В зоне ответственности

| Компонент | Роль |
|-----------|------|
| Клиент (Web) | UI, `getUserMedia`, RTCPeerConnection, WHIP/WHEP SDP-обмен, WebSocket-сигналинг |
| API Gateway / Signaling | HTTP API, WebSocket, аутентификация, лимиты, аудит, прокси к SRS |
| SRS | Медиа-SFU, ICE, маршрутизация RTP; endpoints WHIP/WHEP |
| TURN (обязательно в prod за NAT/моб. сетями) | Relay для ICE, когда прямой UDP невозможен |
| Хранилище | Сессии, комнаты (при необходимости), метрики/логи |

### 1.3. Вне зоны (опционально в v2+)

- Запись эфира, транскрипция, чат, демонстрация экрана как отдельные фичи с отдельными UC.
- Полная замена SRS другим SFU без смены внешнего контракта API (целевой подход — абстракция «медиа-бэкенд»).

## 2. Отличия прототипа от production

| Аспект | Прототип | Production |
|--------|----------|------------|
| Идентификация | Анонимный `peerId` в `sessionStorage` | Учётные записи или одноразовые токены комнаты; привязка сессии к пользователю/роли |
| Доступ к комнатам | Любой, кто знает `roomId` | Политики: публичные/по приглашению/по списку; TTL токенов |
| WebSocket | Без аутентификации | Токен в query/header при upgrade или отдельный handshake после connect |
| CORS | `origin: *` | Явный whitelist origin |
| WHIP/WHEP | Прокси без авторизации | Подписанные URL или одноразовые credentials; rate limit; привязка `peer`/`stream` к токену |
| Состояние | In-memory на одном процессе | Redis (или аналог) для комнат/пиров при горизонтальном масштабировании |
| Надёжность | Один инстанс | Health/readiness, graceful shutdown, рестарты, несколько реплик за LB |
| ICE | Публичный STUN | STUN + **TURN** с выдачей временных credentials |
| Наблюдаемость | Минимальные логи | Структурированные логи, метрики, трассировка, алерты |
| SRS | Один контейнер, ручной `SRS_EIP` | Инфраструктурные шаблоны, автоматический выбор candidate/TURN |

## 3. Акторы и роли

| Актор | Описание |
|-------|----------|
| **Гость** | Участник без постоянного аккаунта (гостевой токен комнаты) |
| **Пользователь** | Зарегистрированный участник (если включена регистрация) |
| **Ведущий / Host** | Создатель комнаты или назначенный модератор: исключение, блокировка публикации |
| **Администратор системы** | Управление инстансами, квотами, ключами TURN, политиками |
| **Сервис (machine)** | CI, мониторинг, вызовы только по service API keys |

Роли задаются в JWT/сессии и проверяются на HTTP и WebSocket.

## 4. Пользовательские сценарии (User Cases)

### UC-01. Создание комнаты

**Актор:** Пользователь или Гость (в зависимости от политики продукта).  
**Предусловия:** Клиент аутентифицирован или запрос анонимный с CAPTCHA/лимитом (политика).  
**Основной поток:**

1. Клиент вызывает `POST /api/v1/rooms` (или получает ссылку от хоста).
2. Сервер создаёт запись комнаты: `room_id`, настройки (max_participants, ttl, запись запрещена и т.д.).
3. Сервер возвращает `room_id` и URL для входа; при гостевом режиме — `join_token` для WebSocket и медиа.

**Постусловия:** Комната существует в хранилище, возможен вход по политике.

**Альтернативы:** A1 — превышена квота комнат на организацию → `429`/`403`. A2 — анонимное создание отключено → `403`.

---

### UC-02. Вход в комнату (лобби)

**Актор:** Участник.  
**Предусловия:** Известны `room_id` и действительный `join_token` (или сессия пользователя с правом входа).  
**Основной поток:**

1. Клиент открывает приложение с `room_id`.
2. Клиент получает ICE-конфигурацию: `GET /api/v1/rtc/ice-config` (STUN/TURN).
3. Клиент устанавливает WebSocket с токеном.
4. После `join` клиент получает `state` со списком текущих издателей.

**Постусловия:** Участник числится в комнате на сигнальном уровне; медиа ещё может быть не запущено.

**Альтернативы:** A1 — комната заполнена → ошибка в ответе на join. A2 — комната закрыта/истекла → `410`/`404`.

---

### UC-03. Начало публикации (микрофон и камера)

**Актор:** Участник.  
**Предусловия:** WebSocket в состоянии joined; разрешена публикация (нет бана; не превышен лимит издателей — если задан).  
**Основной поток:**

1. Клиент запрашивает у пользователя `getUserMedia`.
2. Клиент создаёт `RTCPeerConnection`, формирует offer.
3. Клиент вызывает `POST /api/v1/rtc/whip` с телом SDP и авторизацией (см. API).
4. Сервер валидирует токен, проксирует в SRS, возвращает answer SDP.
5. Клиент отправляет по WebSocket `publishing`.
6. Остальные участники получают `peer-publish` и инициируют WHEP к этому пиру.

**Постусловия:** Поток зарегистрирован в SRS под согласованным `stream_id`; остальные могут подписаться.

**Альтернативы:** A1 — отказ в медиа-разрешениях в браузере — только UI. A2 — SRS недоступен — `502`, клиент показывает retry. A3 — невалидный токен — `401`.

---

### UC-04. Просмотр удалённых участников

**Актор:** Участник.  
**Предусловия:** Известен `peer_id` издателя из сигналинга.  
**Основной поток:**

1. По событию `state` или `peer-publish` клиент создаёт recv-only PC.
2. `POST /api/v1/rtc/whep` с SDP, указывая целевой поток (тот же идентификатор, что у издателя в SRS).
3. Установка remote description, воспроизведение треков.

**Альтернативы:** A1 — издатель остановился — `peer-unpublish` или закрытие PC. A2 — ICE failed — retry с новым offer или сообщение пользователю (TURN).

---

### UC-05. Остановка публикации

**Актор:** Издатель.  
**Основной поток:** Закрытие PC, остановка треков, WebSocket `unpublish`, рассылка `peer-unpublish`.

---

### UC-06. Выход из комнаты / закрытие вкладки

**Актор:** Участник.  
**Основной поток:** WebSocket close; сервер удаляет сокет из комнаты; если был издатель — `peer-unpublish` остальным; освобождение ресурсов SRS (по политике SRS/session timeout).

---

### UC-07. Переподключение WebSocket

**Актор:** Участник.  
**Предусловия:** Токен ещё валиден или refresh выполнен.  
**Основной поток:** Exponential backoff, новое соединение, повторный `join` с тем же `peer_id` или новым (политика: фиксированный peer_id из токена vs клиентский UUID). Синхронизация состояния через `state`.

---

### UC-08. Модерация: запрет публикации

**Актор:** Host.  
**Основной поток:** `POST /api/v1/rooms/{roomId}/participants/{peerId}/mute` (или аналог); сервер помечает участника; при следующем `publishing` — отказ `403`; активную сессию — принудительное уведомление `moderation` по WS и закрытие WHIP на стороне клиента по событию.

---

### UC-09. Исключение участника

**Актор:** Host.  
**Основной поток:** `DELETE /api/v1/rooms/{roomId}/participants/{peerId}`; сервер закрывает WebSocket соединение с кодом `4403` (custom) или отправляет `kicked`, затем закрывает.

---

### UC-10. Наблюдатель (только приём)

**Актор:** Участник.  
**Предусловия:** Роль `viewer` в токене.  
**Основной поток:** Те же WS и WHEP; WHIP запрещён — `403` на `/rtc/whip`.

---

### UC-11. Операционный контроль

**Актор:** Администратор.  
**Потоки:** Просмотр метрик (активные комнаты, RPS, ошибки SRS), `GET /health/live` и `GET /health/ready`, ротация секретов, масштабирование реплик.

---

### UC-12. Клиент за жёстким NAT / мобильная сеть

**Актор:** Участник.  
**Предусловия:** В `ice-config` выдан TURN.  
**Основной поток:** ICE использует relay; медиа проходит через TURN.  
**Критерий успеха:** Соединение устанавливается там, где без TURN было бы `failed`.

## 5. Нефункциональные требования

- **Доступность:** целевой SLA (например 99.9% для API); readiness отделяет зависимости (Redis, SRS).
- **Масштабирование:** stateless HTTP-воркеры; sticky sessions **не** обязательны при shared state в Redis; WebSocket — sticky к инстансу или unified bus (Redis Pub/Sub / NATS) для broadcast между нодами.
- **Безопасность:** TLS везде; секреты в vault; минимальные права токенов; защита от злоупотребления (rate limit по IP и по токену).
- **Задержка:** медиа — путь клиент ↔ SRS (и TURN); сигналинг не в критическом пути после установки сессий.
- **Соответствие:** хранение персональных данных — по политике (логи без сырого SDP при необходимости).

## 6. Идентификаторы и именование потоков SRS

**Соглашение (рекомендуемое для prod):**

- `app`: фиксированное значение из конфига (как в прототипе: `live`) или `live-{tenantId}` при мультитенантности.
- `stream`: уникален в пределах `app`; в прототипе stream = `peerId`. В prod: `stream = f"{roomId}:{publisherSessionId}"` или криптостойкий id, выданный сервером, чтобы нельзя было угадать чужой поток.
- Параметр `eip` (внешний IP/кандидат для SRS) задаётся сервером из конфигурации деплоя, а не клиентом.

## 7. Полная спецификация API

Общие правила:

- Базовый префикс: **`/api/v1`**.
- Заголовок **`Authorization: Bearer <JWT>`** для пользовательских вызовов; для сервис-сервис — **`X-Api-Key`** (отдельный контур).
- Заголовок **`X-Request-Id`** опционально с клиента; иначе генерируется сервером, возвращается в `X-Request-Id`.
- Ошибки — JSON:

```json
{
  "error": {
    "code": "ROOM_FULL",
    "message": "Room has reached max participants",
    "request_id": "uuid"
  }
}
```

### 7.1. Health (без версии или `/api/v1`)

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/health/live` | Процесс жив (для LB) |
| GET | `/health/ready` | Готов принимать трафик: Redis, SRS API reachability |

**Ответ `GET /health/ready` (200):**

```json
{
  "ok": true,
  "checks": {
    "redis": "ok",
    "srs_api": "ok"
  }
}
```

**503** если любая критичная проверка не прошла.

*(Прототип: `GET /api/health` — в prod заменить/расширить.)*

### 7.2. ICE конфигурация

#### `GET /api/v1/rtc/ice-config`

**Назначение:** Выдать `iceServers` для `RTCPeerConnection`.

**Заголовки:** `Authorization: Bearer <join_or_user_jwt>`.

**Ответ 200:**

```json
{
  "iceServers": [
    { "urls": "stun:stun.example.com:3478" },
    {
      "urls": "turn:turn.example.com:3478",
      "username": "temp-user",
      "credential": "temp-secret",
      "credentialType": "password"
    }
  ],
  "ttlSeconds": 86400
}
```

**401** — невалидный токен.  
**Примечание:** TURN credentials — краткоживущие (как у coturn REST API или статический HMAC).

### 7.5. WebRTC: WHIP (публикация)

#### `POST /api/v1/rtc/whip`

**Назначение:** Установить исходящую WebRTC-сессию к SRS (как сейчас прокси к `/rtc/v1/whip/`).

**Заголовки:**

- `Authorization: Bearer <token>` — обязателен; claims содержат `roomId`, `peerId`, `role`, `streamId` (или право публиковать в вычисляемый stream).
- `Content-Type: application/sdp`

**Тело:** raw SDP offer (binary/text).

**Поведение сервера:**

1. Валидация JWT и роли (не `viewer`).
2. Сопоставление `stream` для SRS = из токена (не из query string от клиента).
3. Прокси POST на SRS:  
   `{SRS_HTTP}/rtc/v1/whip/?app={app}&stream={streamId}&eip={SRS_EIP}`  
   (как в прототипе, но `stream`/`app` с сервера).

**Ответ 201 или 200:** `Content-Type: application/sdp` — answer SDP (как у SRS).

**Ошибки:**

| HTTP | code | Когда |
|------|------|--------|
| 400 | `INVALID_SDP` | Невалидное тело |
| 401 | `UNAUTHORIZED` | Нет/плохой токен |
| 403 | `FORBIDDEN` | viewer или revoke-publish |
| 429 | `RATE_LIMITED` | Слишком частые negotiate |
| 502 | `SRS_ERROR` | SRS недоступен или ошибка upstream |

---

### 7.6. WebRTC: WHEP (подписка)

#### `POST /api/v1/rtc/whep`

**Назначение:** Подписаться на поток другого участника.

**Заголовки:** `Authorization: Bearer <token>` (тот же join-токен комнаты).

**Тело:** raw SDP offer (recvonly).

**Параметры (query или предпочтительно одно тело JSON в отдельном preflight — ниже вариант с query):**

Рекомендуется **не** доверять произвольному `peer` из query без проверки. Варианты:

- **A (строгий):** В теле второго запроса не используется; сервер принимает только `targetPeerId` в **подписанном** теле или выдаёт одноразовый `subscriptionId` через REST:

#### `POST /api/v1/rtc/subscriptions`

**Тело:**

```json
{ "targetPeerId": "pr_01..." }
```

**Ответ 201:**

```json
{ "subscriptionId": "sub_01...", "expiresIn": 60 }
```

Затем клиент вызывает:

#### `POST /api/v1/rtc/whep?subscriptionId=sub_01...`

С SDP; сервер проверяет `subscriptionId` и строит URL к SRS для нужного `stream`.

- **B (упрощённый internal):** `POST /api/v1/rtc/whep?targetPeerId=` + проверка, что `targetPeerId` в той же `roomId`, что в JWT.

**Ответ:** `application/sdp` — answer.

**Ошибки:** 401, 403 (чужая комната / подписка на себя запрещена политикой), 404 (нет такого издателя), 502.

### 7.7. WebSocket: подключение

#### URL

`wss://{host}/api/v1/ws?token={JWT}`

Альтернатива: подключение без token + первое сообщение `auth` (хуже для прокси; предпочтительно token в query или `Sec-WebSocket-Protocol`).

**Протокол:** JSON-текстовые сообщения, UTF-8.

**Версионирование:** Первое сообщение от клиента после open — опционально `hello`; сервер отвечает `hello-ack` с `protocolVersion: 1`.

---

### 7.8. WebSocket: сообщения клиента → сервер

| Поле `t` | Поля | Описание |
|----------|------|----------|
| `join` | `roomId` (опционально, если зашито в JWT), `clientCapabilities` (опционально) | Регистрация сокета в комнате; `peerId` **только из JWT**, не из тела (игнорировать или запрещать). |
| `publishing` | — | Уведомить комнату, что WHIP-сессия активна (или подтвердить после server-side verify). |
| `unpublish` | — | Снять публикацию с сигнального учёта. |
| `ping` | `nonce` | Сервер отвечает `pong`. |

**Прототип сейчас:** `join` с `roomId` и `peerId` с клиента — в prod **убрать доверие к client `peerId`**.

---

### 7.9. WebSocket: сообщения сервер → клиент

| Поле `t` | Поля | Описание |
|----------|------|----------|
| `hello-ack` | `protocolVersion`, `serverTime` | После подключения |
| `state` | `publishers: string[]` | Список `peerId` активных издателей (кроме получателя, по политике) |
| `peer-publish` | `peerId` | Новый издатель |
| `peer-unpublish` | `peerId` | Издатель остановился |
| `participant-joined` | `peerId`, `displayName?` | Опционально |
| `participant-left` | `peerId` | Опционально |
| `moderation` | `action`, `reason` | Например запрет публикации |
| `room_closed` | `reason` | Комната закрыта |
| `error` | `code`, `message` | Ошибка протокола |
| `pong` | `nonce` | Ответ на ping |

---

### 7.10. Коды ошибок WebSocket (закрытие)

| Код | Причина |
|-----|---------|
| 4001 | Невалидный токен |
| 4002 | Истёк токен |
| 4003 | Комната не найдена / закрыта |
| 4004 | Кик модератором |

### 7.11. Обратная совместимость с прототипом (миграция)

Для переходного периода допускается префикс **`/api/legacy/`**:

- `POST /api/legacy/rtc/whip?peer=` — как сейчас, за фичефлагом и только в dev/staging.

В production трафик должен идти на **`/api/v1`** с токенами.

## 9. Данные для хранения (минимум)

- **Room:** id, tenant/org (если есть), created_at, expires_at, settings, created_by.
- **ParticipantSession:** room_id, peer_id, display_name, role, joined_at, connection_id (WS).
- **Ephemeral:** subscription ids для WHEP (TTL 60s), TURN credentials TTL.

In-memory Maps из прототипа заменяются на **Redis**:

- Ключи вида `room:{id}:sockets` (set connection ids), `room:{id}:publishers` (set peer ids).
- Pub/Sub канал `room:{id}` для broadcast между инстансами Node.
